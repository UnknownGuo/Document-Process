# 自动化文档处理微创业项目 - 工作流方案 (PLAN V2.0 混合架构版)

## 核心架构：多智能体质检 + 无损排版引擎
本系统基于“大模型智能赋能软件工程”的理念，融合了 `word-MCP-agent` 的多智能体协作思想与独创的“底层无损格式注入技术”。致力于打造一套高内聚、低耦合，且具备自我审查能力的工业级文档自动化流水线。

---

## 完整业务流转标准操作程序 (SOP)

### 阶段一：业务接入与模板配置（用户端 - Windows）
1. **自然语言占位**：用户打开空白的 Word 业务模板（如：教案、周报、合同）。
2. **【中文指令】注入**：抛弃反人类的代码标签（如 `{{ title }}`）。在需要 AI 生成的区域，直接使用中文字符 `【】` 包裹具体指令。
   * *示例：`【生成3个极简学习目标】`、`【根据上下文生成预设及补救措施，并在末尾加上达成度⭐⭐⭐】`。*
3. **排版继承锁定**：用户直接对 `【中文指令】` 设置最终期望的排版格式（如：黑体、加粗、居中、红色字体）。系统底层引擎在后续渲染时将 100% 继承这些样式，绝不破坏原有 XML 结构，彻底杜绝“文档损坏/无法读取”报错。

### 阶段二：系统意图对齐与单篇验证（双端协作）
1. **解析与初稿生成**：系统读取带有 `【】` 指令的模板，利用大模型（如 Gemini 2.5 Pro）初步生成测试样稿（如 `_v1.docx`）。
2. **人工微调闭环 (Modify Loop)**：
   * 用户在 Windows 环境下人工查阅样稿。若发现逻辑、排版或细节瑕疵，用户直接在样稿上进行修改并另存为 `_modify.docx`（或开启 Word 修订模式保存）。
   * 系统通过底层脚本自动比对 `_v1.docx` 与 `_modify.docx` 的段落/表格/修订标签（`w:ins`, `w:del`）。
   * 系统自动提取用户的**隐式修改意图**（例如：“目标只保留2条”、“动作必须分行且末尾单独加星星”）。
3. **确认与固化**：系统向用户汇报提取出的意图。用户确认后，系统将这些意图固化为**“严格质检约束条件 (Constraints)”**，并写入该 Case 的底层 Prompt 中。

### 阶段三：混合智能管线批量生成（系统端 - Linux/WSL）
一旦意图对齐完毕，进入无人值守的批量生成阶段。系统启动核心引擎 `llm_engine.py` 中的**“多智能体协作管线 (Multi-Agent Pipeline)”**：
1. **撰写者 (Writer Agent)**：根据上下文指令和模板位置，负责起草纯文本草稿。
2. **审查者 (Reviewer Agent) [核心防线]**：
   * 充当无情的质检员，拿着阶段二固化的“约束条件 (Constraints)”对草稿进行严格打分 (1-10分)。
   * **自动纠错返工**：如果评分低于 8 分或违反硬性排版要求（如使用了不该使用的 1. 2. 自动编号、字数超标、少打星星），Reviewer 将直接拒绝，并把修改意见反馈给 Writer 要求重写。
   * 此循环最多执行 3 次，直至产出免检的“完美文稿”。
3. **无损注入器 (Surgical Injector)**：将满分文稿通过纯字符串安全替换技术，严丝合缝地填回原始模板的 `【】` 中。

### 阶段四：交付归档
* 成果物将被自动输出至指定的业务共享挂载盘。
* **命名规范**：`A:\AI_Studio\Auto-docs\xxx\output\生成文件名_v[n]_AutoReview_[YYYYMMDD].docx`。

## 架构演进日志
- **V1.0**: 基于 `docxtpl` 和 `python-docx` 的暴力 XML 替换，导致文档易损坏，格式易丢失。
- **V2.0**: 引入“自然语言 `【指令】` 占位”、“基于修订模式的人工意图对齐机制”，以及“Writer + Reviewer 多智能体审查循环”。实现高质量、高稳定性的商业级交付。
